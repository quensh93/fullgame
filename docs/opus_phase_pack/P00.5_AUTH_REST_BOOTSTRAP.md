# P00.5 - Auth REST Bootstrap (Paste to Opus)

```md
Phase 00.5 only: implement auth bootstrap before WS-v3 phases.

Goal:
- finalize REST signup/login so websocket AUTH always uses a fresh valid token.

Requirements:
1. Implement REST auth calls with typed payloads:
   - `POST /api/auth/signup` body: `{ username, email, password }`
   - `POST /api/auth/login` body: `{ email, password }`
   - `POST /api/auth/logout` body: `{ refreshToken }` when available
2. Normalize auth response shape:
   - read access token from `accessToken` fallback `token`
   - read optional `refreshToken`
   - parse user profile from response (direct fields or nested `user`)
3. Persist auth session:
   - store access + refresh token
   - clear both tokens on logout
   - clear both tokens on auth-invalidating websocket errors
4. Auth-to-WS handoff:
   - after successful signup/login, save user+tokens then connect/reconnect websocket manager
   - websocket AUTH send path must always read latest stored access token
5. Error policy alignment:
   - REST `401/403` => signed-out state + login-required UX
   - websocket `AUTH_EXPIRED`, `TOKEN_REVOKED`, `INVALID_TOKEN` => clear auth, disconnect socket, route to login
6. UI binding:
   - login/register forms use auth controller methods
   - loading state + button disable while pending
   - map common errors: invalid credentials, duplicate email/username, network/server failure

Validation checklist:
- signup success -> token persisted -> websocket connect -> `AUTH_SUCCESS`
- login success -> token persisted -> websocket connect -> `AUTH_SUCCESS`
- invalid credentials -> no authenticated websocket flow
- logout -> tokens cleared + websocket disconnected

Output format:
1. changed files
2. exact request/response shapes handled
3. auth-to-ws handoff sequence
4. remaining risks before Phase 01
```
